# 🇬🇧 Observability SRS / 🇷🇺 Техническое задание на наблюдаемость

## 🇬🇧 Overview / 🇷🇺 Обзор

The platform must provide comprehensive observability of service‑level metrics: collection, processing, calculation of SLI, comparison with SLO, and exposure via GraphQL.
Платформа должна обеспечивать полную наблюдаемость метрик уровня обслуживания: сбор, обработку, расчёт SLI, сравнение с SLO и предоставление через GraphQL.

## 🇬🇧 Functional Requirements / 🇷🇺 Функциональные требования

### FR‑OBS‑001: Приём метрик
Система должна принимать метрики от внешних источников через Gateway. Метрики должны включать имя, значение, метки (labels), временную метку и идентификатор источника.

### FR‑OBS‑002: Многостадийная обработка
Метрики должны проходить через Pipeline, который выполняет нормализацию, фильтрацию, агрегацию и запись.

### FR‑OBS‑003: Хранение агрегатов
Агрегированные метрики должны сохраняться в PostgreSQL для последующего анализа.

### FR‑OBS‑004: GraphQL API
Должен быть реализован GraphQL API для запроса:
- текущих значений метрик,
- статуса SLO (ok, warning, breach),
- истории соблюдения SLO.

### FR‑OBS‑005: Расчёт SLI
На основе агрегированных данных должны вычисляться следующие SLI:
- **доступность** (доля успешных ответов),
- **задержка** (p50, p95, p99),
- **частота ошибок** (доля ответов с ошибками),
- **пропускная способность** (запросы в секунду).

### FR‑OBS‑006: Проверка SLO
Система должна поддерживать определение целевых SLO и отображать статус их соблюдения. При выходе за пределы допустимых значений должен фиксироваться факт нарушения (breach).

### FR‑OBS‑007: Бакеты ошибок
Для детального анализа ошибок система должна поддерживать разбиение по типам ошибок (например, по HTTP‑статусам: 400, 500, таймауты) и сохранять их распределение во времени.

### FR‑OBS‑008: Проверка частоты запросов
Система должна отслеживать частоту входящих запросов (RPS) и сравнивать её с пороговыми значениями для обнаружения аномальных всплесков или падений трафика.

### FR‑OBS‑009: Пороги скорости ответа
Должна быть возможность задавать пороговые значения для задержки (например, p95 ≤ 200 мс). При превышении порога система должна фиксировать нарушение.

## 🇬🇧 Non‑Functional Requirements / 🇷🇺 Нефункциональные требования

- **NFR‑OBS‑001**: Pipeline должен поддерживать backpressure для предотвращения переполнения памяти.
- **NFR‑OBS‑002**: Все компоненты должны поддерживать graceful shutdown.
- **NFR‑OBS‑003**: GraphQL API должен отвечать менее чем за 1 секунду при 100 одновременных запросах.
- **NFR‑OBS‑004**: Расчёт SLI должен выполняться в реальном времени с задержкой не более 1 минуты.
